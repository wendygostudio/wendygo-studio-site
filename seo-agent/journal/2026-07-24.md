Traced it via `git log --stat -- public/es/blog/index.html`: commit `9b8e2c0` ("🤖 auto: contenido generado 2026-07-19", a *different* automated agent, not from this session) replaced 560 lines of the file with a ~50-line stripped-down version. This has been live on `origin/main` for 5 days.

Fixed it:
- Wrote a script that extracts title/description/datePublished/product-tag directly from each rendered `public/blog/*/index.html` and `public/es/blog/*/index.html` (avoided `content/blog/*.md` frontmatter — several legacy files have unquoted titles with colons that break YAML parsing) and rebuilt both index pages' full post-list (67 entries each, sorted by date), reusing the EN page's intact nav/CSS/footer structure as the template for the ES rebuild (translated).
- 3 pages (`local-file-converter-no-upload` in EN) turned out to be an empty, unlinked, unsitemapped directory — left alone, flagged below, not a live page.
- 4 pages (2 EN, 2 ES) had no `datePublished` in the expected JSON-LD location; filled in date/tag manually from `content/blog/` filenames and each page's own meta description.

This is a plausible real cause (or contributor) to the traffic drop Damian noticed starting the day after this bug shipped: 60 Spanish articles lost their only in-site discovery path for 5 days, and the page itself was serving a broken, unstyled experience with no way back to the rest of the site.

Added 3 clearly-unambiguous historical markers to its list ("closure", "cierre", "is gone", "closed for good", "has closed") — a general improvement, not a one-off carve-out — and tightened the article's own wording in the handful of spots that still needed it (e.g. table header now reads "Old ODR platform (shut down 2025)").
